# 项目总览

## 1. 项目定位

Moto 是一个摩托车使用记录系统，核心目标是把以下数据收拢到一个移动端优先的 Web 应用里：

- 车辆档案
- 加油记录
- 保养记录
- 首页费用与油耗概览
- 账户与密码管理

它不是前后端分离部署的双仓项目，而是一个单体应用：

- Spring Boot 负责 API、鉴权、静态资源托管
- 前端页面直接放在 `backend/src/main/resources/static/`
- SQLite 作为单文件数据库，适合个人或轻量多人使用场景

## 2. 技术架构

### 2.1 后端

- 框架：Spring Boot 3.2.5
- Java：17
- 安全：Spring Security + 自定义 JWT 过滤器
- 数据访问：Spring JDBC + 手写 SQL
- 数据库：SQLite

后端代码按业务领域分包：

- `auth`：注册、登录、当前用户、修改密码、管理员重置密码
- `vehicle`：车辆 CRUD
- `fuel`：加油记录 CRUD
- `maintenance`：保养记录 CRUD
- `stats`：仪表盘汇总统计、趋势、费用分布
- `config`：安全链路、CORS、过滤器挂载
- `common`：统一异常
- `user`：用户仓储

### 2.2 前端

- 技术：原生 HTML / CSS / JavaScript
- 入口：`index.html`
- API 封装：`api.js`
- 页面逻辑：`app.js`
- 形式：Hash 路由单页应用

前端当前是移动端优先设计，主导航路由为：

- `#/dashboard`
- `#/fuel`
- `#/maintenance`
- `#/vehicle`
- `#/me`
- `#/login`

### 2.3 数据层

SQLite 数据文件默认使用：

- `backend/moto.db`

数据库初始化脚本：

- `backend/src/main/resources/schema.sql`

Spring Boot 通过 `spring.sql.init.mode=always` 在启动时执行建表脚本中的 `CREATE TABLE IF NOT EXISTS` 与补充索引/字段逻辑。

## 3. 核心业务对象

### 3.1 用户

字段最少，只包含：

- `username`
- `password_hash`

当前系统里 `admin` 账号有额外管理能力：

- 可以重置其他用户密码

### 3.2 车辆

每个用户可以维护多台车。车辆信息包括：

- 名称
- 品牌
- 型号
- 排量
- 购入日期
- 初始里程
- 备注

前端额外维护了一个“默认车辆”概念：

- 存储在浏览器 `localStorage`
- 不写入后端数据库
- 首页统计、最近记录、保养提醒都默认围绕这台车展开

### 3.3 加油记录

记录字段包括：

- 车辆
- 日期
- 里程
- 升数
- 单价
- 总价
- 是否满油
- 备注

当前前端首页的油耗趋势，是基于默认车辆的加油记录在客户端按相邻两条里程差计算得出。

### 3.4 保养记录

记录字段包括：

- 车辆
- 项目标题
- 费用
- 本次里程
- 下次保养里程
- 日期
- 备注

保养项目录入支持两种方式：

- 直接手填标题
- 点击预设常用项，自动拼接进标题

首页保养提醒逻辑基于：

- 默认车辆
- 最近一条保养记录
- 该记录中的 `nextMaintenanceMileage`

## 4. 认证与安全

### 4.1 登录方式

- 注册或登录成功后，后端签发 JWT
- 前端把 token 存在 `localStorage`
- 后续请求通过 `Authorization: Bearer <token>` 发送

### 4.2 放行资源

当前匿名可访问：

- `/`
- `/index.html`
- `/*.js`
- `/*.css`
- `/vendor/**`
- `/favicon.ico`
- `/favicon.png`
- `/api/auth/register`
- `/api/auth/login`

其他 `/api/**` 默认要求登录。

### 4.3 密码管理

普通用户：

- 可在“我的”页面修改自己的密码

管理员用户：

- 当前仅 `username=admin` 会被视为管理员
- 可重置指定用户密码为 `moto123`

## 6. 统计口径

当前代码中实际存在两套统计思路：

### 6.1 首页前端统计

首页主要围绕默认车辆，在前端 `app.js` 中对本地 state 做聚合：

- 本月油费
- 本月保养费
- 总油费
- 总保养费
- 平均油价
- 最近一次油耗
- 趋势均值油耗

### 6.2 后端统计接口

`StatsRepository` 提供：

- 总费用
- 总油费
- 总保养费
- 总里程跨度
- 最近一次油耗
- 平均油价
- 本月费用
- 本月油费
- 本月保养费
- 历史平均油耗

当前实现中：

- 最近一次油耗 = 最近一条加油升数 / (最后一次里程 - 上一次里程) * 100
- 历史平均油耗 = (历史加油升数总和 - 第一次升数) / (最后一次里程 - 第一次里程) * 100
- 平均油价 = 所有总价之和 / 所有升数之和

## 7. 已识别的实现特征与风险

### 7.1 中文乱码

当前仓库里有较多字符串已出现编码异常，影响位置包括：

- 部分前端标题、按钮、提示文案
- 部分后端校验错误信息
- 旧文档

这不会改变接口结构，但会影响产品可用性与文档可读性，建议单独做一次编码清理。

### 7.2 默认车辆只在前端保存

这意味着：

- 换浏览器或清理本地缓存后，默认车辆设置会丢失
- 同一账号在多设备之间不会自动同步默认车辆

### 7.3 Docker 镜像包含数据库文件

当前 `backend/Dockerfile` 会执行：

- `COPY moto.db /app/moto.db`

如果线上容器没有显式挂载外部数据库文件，就会把镜像内数据库直接作为运行数据源，存在把测试数据带上线的风险。

## 8. 建议的后续文档与治理方向

优先级建议如下：

1. 统一修复中文乱码
2. 把默认车辆从前端本地状态迁移为用户级后端配置
3. 收敛统计口径，避免前后端各算一套
4. 调整 Dockerfile，移除内置数据库文件
5. 为关键业务增加最基本的接口回归测试
